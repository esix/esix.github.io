В детстве на ПК XT я играл в одну игру — ПакМен, сделанный в Киеве в 1990 году командой Free Masters. Не оригинальный Pac-Man от Namco, а советский клон — с русскими надписями, «кровавыми жуками» вместо призраков и своим характером. Решил её портировать.

Бинарник нашёлся — `PACKMAN.EXE`, 42 килобайта. MZ DOS executable, запускается в DOSBox.

Проблема в том, что исходников нет. Только бинарь.

## Дизассемблирование

Для декомпиляции использовал [Reko Decompiler](https://github.com/uxmal/reko). Он выдал:

- 6944-строчный листинг на x86 ASM
- приблизительный декомпилированный C

C-вариант Reko-а читается удобнее, но местами врёт — переменные без имён, логика угадана неточно. Поэтому правило: **если ASM и C расходятся, верим ASM**.

Дальше — ручной разбор ключевых функций: инициализация уровней, движение сущностей, AI врагов, обработка клавиатуры, коллизии, подсчёт очков.

## BGI и CGA

Игра написана на Turbo C и использует **BGI (Borland Graphics Interface)** — стандартную графическую библиотеку для DOS-программ от Borland. В бинаре прямо прописана строка `"CGA Device Driver 2.00 — Mar 21 1988"` — встроенный BGI-драйвер для CGA.

CGA-режим: 320×200 пикселей, 4 цвета из фиксированной палитры. Никакого antialiasing, никаких полутонов — чистый пиксель.

BGI предоставлял набор функций для рисования:
- `outtextxy` — вывод текста в нужной точке
- `bar` — закрашенный прямоугольник
- `putimage` — спрайт из памяти
- `settextstyle` — выбор шрифта: Gothic (#4) для заголовка, Triplex (#1) для текста меню
- `setcolor`, `setfillstyle` — управление цветом и заливкой

Спрайты в игре — 2bpp CGA-формат: каждый байт хранит 4 пикселя, по 2 бита на пиксель. Каждый спрайт — 16×14 px, 56 байт. Всего 29 спрайтов: стены лабиринта, точки, кадры анимации ПакМена и врагов.

## Что внутри

Вытащил из бинаря всё что можно — скриптом на Node.js, зная формулу смещения: `file_offset = DS_offset + 0x400`.

Интересные детали, найденные в ASM:

**Количество жизней** — вычисляется как `0x23 / 7 = 5`. В CLAUDE.md изначально было написано «3 жизни, из `0x23 >> 4`» — неверно. Настоящая инструкция `idiv 7`, результат 5.

**Скорость** — хранится инвертированно: нажатие `0` даёт внутреннее значение 9 (медленно), нажатие `9` — 0 (быстро). Формула: `speed = '9' - key`. Отображаемое значение `9 - speed`.

**Движение асимметрично**: горизонтально 4 пикселя за шаг, вертикально 2. Это не баг — так в ASM.

**AI врагов**: при совпадении строки или столбца с ПакМеном — преследование, иначе — следование предопределённому паттерну из 16 направлений. Три разных паттерна, назначаются случайно при старте уровня.

**Нет режима испуга**. Никакого. Столкновение с врагом — всегда смерть. «Кровавые жуки» — не призраки, с ними не договориться.

**«Победа !»** показывается только после прохождения всех 15 уровней — не после каждого. Между уровнями игра просто сразу стартует следующий.

**Уровень 13** — это отдельная история.

## Уровень 13

В детстве я дошёл до 13-го уровня один-единственный раз. И обнаружил, что он непроходим — ПакМен заперт в маленьком участке лабиринта, где всего две точки. Съел их — и всё, выхода нет. Жуки бегают снаружи, а ты стоишь.

Это было разочарование. Пройти 12 уровней сложной игры без режима испуга, без power-up'ов — и упереться в баг.

Одной из целей этого порта было проверить: правда ли 13-й уровень сломан, или я что-то не так помню? Ответ нашёлся в данных.

Данные уровней хранятся начиная с `DS:0x6FFC` в исполняемом файле. Каждый уровень — 480 байт: 20 столбцов × 12 строк × 2 байта на тайл. 13-й уровень (индекс 12) — по смещению `DS:0x867C` (файловое `0x8A7C`).

Тайлы стен лабиринта:

| ID | Символ | Значение |
|----|--------|----------|
| 0x02 | `╭` | угол: верхний левый |
| 0x07 | `╮` | угол: верхний правый |
| 0x08 | `╯` | угол: нижний правый |
| 0x09 | `╰` | угол: нижний левый |
| 0x0A | `│` | вертикальная стена |
| 0x0B | `═` | горизонтальная стена |
| 0x00 | ` ` | пустое / проходимое |
| 0x01 | `·` | точка (2 очка) |

Карта уровня из бинарника:

```
╭══════════════════╮     строка 0
│··················│     строка 1
╰╮·╭╮·╭╮·╭╮·╭╮·╭╮·╭╯    строка 2
 │·╰╯·╰╯·╰╯·╰╯·╰╯·╰╮   строка 3  ← col 0 = 0x00 (баг!)
╭╯·················│    строка 4
│·╭═╮·╭╮·╭╮·╭╮·╭╮·╭╯    строка 5
│ᗧ╰═╯·╰╯·╰╯·╰╯·╰╯·╰╮   строка 6
╰╮·················│    строка 7
 │·╭╮·╭╮·╭╮·╭╮·╭╮·╭╯   строка 8  ← col 0 = 0x00 (баг!)
╭╯·╰╯·╰╯·╰╯·╰╯·╰╯·╰╮   строка 9
│··················│    строка 10
╰══════════════════╯    строка 11
```

Видите пробелы в начале строк 3 и 8? Это тайл `0x00` — «пустое место», проходимое для ПакМена. Но визуально стены рисуются только для тайлов стен. В результате в левой стене образуются дыры — но не проходы, а **разрывы сетки**: ПакМен не может через них пройти, потому что движется только по прямым линиям внутри сетки. Лабиринт разваливается на изолированные секции.

![Уровень 13 — ПакМен заперт](13th-level.png)

По смещениям `0x8AF4` и `0x8BBC` в файле: вместо `0x0A` (вертикальная стена `│`) записано `0x00` (пусто). Ещё одна подсказка: один из внутренних прямоугольников лабиринта имеет размер 3×2, тогда как все остальные — 2×2. Когда я привёл его к 2×2, замысел авторов стал очевиден — уровень должен был быть полностью симметричным.

Баг, скорее всего, — ошибка при ручном вводе карты уровня. В экстракторе я его исправил: восстановил угловые тайлы в левой стене и открыл проходы между горизонтальными полосами лабиринта, восстановив симметрию с правой стороной. Уровень стал проходимым — детская мечта пройти все 15 уровней наконец сбылась.

![Уровень 13 — после исправления](13th-level-fixed.png)

## Порт на JavaScript

Никакого билд-степа — пять plain JS файлов, `index.html` открывается прямо в браузере.

```
data.js      — уровни, спрайты, палитры (генерируется из EXE)
input.js     — состояние клавиатуры
sound.js     — эмуляция PC-спикера через Web Audio API
renderer.js  — 2bpp декодер спрайтов, Canvas 2D
game.js      — стейт-машина, движение, AI, коллизии
```

Спрайты декодируются при старте из сырых байт в `ImageData` — один раз на палитру. Текст меню в оригинале закодирован в CP866, я его честно вытащил и записал в UTF-8.

Звуки — квадратные волны через `OscillatorNode`, приближение к PC-спикеру.
